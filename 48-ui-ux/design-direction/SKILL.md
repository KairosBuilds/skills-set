# Design Direction

Build it with a point of view. If it could have been generated by a default prompt, it is not good enough.

## Outcome Contract

- Outcome: a usable interface or visual fix with a clear point of view.
- Done when: the real rendered surface has been checked against the user's visual goal and relevant viewport states.
- Evidence: screenshots, rendered UI, source components, design tokens, accessibility constraints.
- Output: the implemented visual change or a precise visual review.

## Lock the Direction First

Before writing any code, establish the direction:

1. Who uses this, and in what context?
2. What is the aesthetic direction? Name it precisely (e.g., "dense editorial," "raw terminal," "warm brutalist"). "Clean and modern" is not a direction.
3. What is the design signature? A typeface, color system, motion, or asymmetric layout.
4. What are the hard constraints? Framework, bundle size, contrast minimums, keyboard accessibility.
5. What is the signature micro-interaction? Scale on press, staggered reveal, or contextual animation.

## Visual Quick-Fix Mode

For narrow visual repairs with concrete symptoms (overflow, misalignment, spacing, contrast):

1. Read the current UI evidence: screenshot, rendered page, or component.
2. Name the exact visual defect in one sentence.
3. Make the smallest material, geometry, spacing, contrast, or typography change that fixes it.
4. Verify in the real running surface. Check long words, localized strings, and compact states.

## Screenshot Iteration Mode

Activate when the user sends a screenshot with a visual complaint.

1. State the problem in one sentence: what specifically looks wrong.
2. Wait for user to confirm diagnosis before touching code.
3. Find the responsible code: grep for component name, read actual file.
4. Apply minimal fix. For existing products, adjust material/opacity, geometry, spacing, typography before redesigning.
5. Verify result at desktop width and 375px mobile width.
6. Ask the user to verify in the browser.

## Hard Rules

- Adjacent nested surfaces must differ visually (background step >= 4% lightness difference or shadow min 0 1px 3px rgba(0,0,0,0.10))
- Test long words and localized strings before handoff, especially in buttons, tabs, nav, and compact cards
- `backdrop-filter` is expensive on low-power devices — name the tradeoff
- No em-dash in any output

## Output

```
Aesthetic direction: [named and justified in 2-3 sentences]
Non-obvious choices: [typeface, color decisions, layout logic]
Viewport verification: [desktop OK, mobile OK, issues if any]
```
