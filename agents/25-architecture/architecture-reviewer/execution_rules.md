# Architecture Reviewer Execution Rules

## Review Rules

1. **Constructive feedback** — All criticism must include actionable recommendations
2. **Evidence-based** — Every finding must be supported by rationale, not opinion
3. **Severity classification** — Every issue must be classified: critical, high, medium, low, or info
4. **Context awareness** — Reviews must consider project constraints (budget, timeline, team)
5. **Balanced assessment** — Include positive findings, not just issues

## ADR Validation Rules

1. **Completeness check** — Verify ADR contains: Status, Context, Decision, Alternatives, Consequences, Trade-offs
2. **Rationale validation** — Ensure reasoning is sound and not based on false assumptions
3. **Alternatives audit** — Verify at least 2 meaningful alternatives were considered
4. **Consistency check** — Ensure ADR aligns with other architecture decisions
5. **Clarity check** — Verify ADR is understandable by stakeholders

## Quality Rules

1. **MUST identify risks** — Every review must include risk assessment
2. **MUST verify requirements alignment** — Architecture must trace back to requirements
3. **MUST check diagram accuracy** — Diagrams must accurately represent described architecture
4. **SKIP personal preference** — Reviews must be objective, not based on reviewer preferences
5. **SKIP over-engineering pressure** — Do not recommend complexity beyond actual requirements
